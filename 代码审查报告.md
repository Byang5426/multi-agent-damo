### 1. Trace 日志从未持久化到数据库，`/trace` 接口永远返回空

**位置**: [workflow.py](src/multi_agent/graph/workflow.py) 全文 + [pg_store.py#L284-L306](src/multi_agent/store/pg_store.py)

**问题**: 工作流各节点（`gateway_route`、`instant_handler`、`project_init`、`worker_execute`、`pm_review`、`handle_failure`）都生成了 `TraceEntry` 并累积到状态的 `trace_logs` 字段，但**没有任何节点调用 `store.save_trace()`**。`PgStore.save_trace()` 方法已实现但从未被调用。最终 trace 数据在请求结束后被丢弃，`GET /api/v1/projects/{project_id}/trace` 接口**永远返回空列表**，系统可观测性形同虚设。

**修复**: 在每个生成 trace 的节点函数中增加持久化调用，例如在 `worker_execute` 末尾：

```python
store: PgStore = state.get("_store")
if store:
    await store.save_trace(trace)
```

建议在 `gateway_route`、`worker_execute`、`pm_review`、`handle_failure` 等所有生成 trace 的节点中添加。

### 2. `handle_failure` 中的状态机转换会导致 ValueError 运行时崩溃

**位置**: [workflow.py#L297-L306](src/multi_agent/graph/workflow.py)

**问题**: 两种场景会触发非法状态转换，导致 `ValueError` 异常并可能中断整个工作流：

- **场景 A — PM 决定 "abort"**: 任务从 `worker_execute` 进入 `handle_failure`，状态为 `FAILED`。代码执行 `task.transition(TaskStatus.FAILED)`，即 `FAILED → FAILED`。但 `VALID_TRANSITIONS[FAILED] = {TODO, HUMAN_PENDING}`，不包含 FAILED 本身，**抛出 ValueError**。
- **场景 B — PM 在 max_retries 路径决定 "retry"**: 任务从 `pm_review`（max_retries 分支）进入 `handle_failure`，状态为 `TODO`（刚被 PM 拒绝回退到 TODO）。代码执行 `task.transition(TaskStatus.TODO)`，即 `TODO → TODO`。但 `VALID_TRANSITIONS[TODO] = {DOING, FAILED}`，不包含 TODO，**同样抛出 ValueError**。

**修复**:

```python
if decision.action == "retry":
    if task.status != TaskStatus.TODO:
        task.transition(TaskStatus.TODO, updated_by="pm")
    task.retry_count += 1
    logger.info("PM decided to retry task '%s'", task.task_id)
elif decision.action == "escalate_to_human":
    if task.status != TaskStatus.HUMAN_PENDING:
        task.transition(TaskStatus.HUMAN_PENDING, updated_by="pm")
    logger.info("PM escalated task '%s' to human", task.task_id)
else:  # abort
    if task.status not in (TaskStatus.FAILED, TaskStatus.DONE):
        task.transition(TaskStatus.FAILED, updated_by="pm")
    logger.info("PM aborted task '%s'", task.task_id)
```

### 3. Gateway 路由 Prompt 指示 LLM 返回 "scheduled"，但代码静默将其当作 "instant" 处理

**位置**: [workflow.py#L359-L368](src/multi_agent/graph/workflow.py) + [prompts.py#L171-L174](src/multi_agent/defaults/prompts.py)

**问题**: `route_after_gateway` 函数只检查 `"blocked"` 和 `"project"`，其他所有值都 fallback 到 `"instant"`。但路由 LLM Prompt 明确告知 LLM 可以返回 `"scheduled"` 路由：

```
"scheduled" — 定时任务：周期性/定时执行的任务（MVP阶段暂不支持，但仍需正确分类）。
{"route": "instant|project|scheduled", ...}
```

当 LLM 返回 `"scheduled"` 时，系统静默将其作为即时任务执行，用户不会收到任何"暂不支持"的提示，而是得到一个可能不合理的即时任务结果。

**修复**:

```python
def route_after_gateway(state: WorkflowState) -> str:
    decision = state.get("route_decision", {})
    route = decision.get("route", "instant")
    if route == "blocked":
        return "blocked"
    elif route == "project":
        return "project"
    elif route == "scheduled":
        return "blocked"  # 返回友好提示"暂不支持"
    else:
        return "instant"
```

同时更新 routing prompt 说明 scheduled 当前会返回"暂不支持"的提示，或者增加一个 `scheduled_handler` 节点返回友好提示。

### 4. `retry_count` 被双重递增，实际重试次数仅为预期的一半

**位置**: [workflow.py#L263-L265](src/multi_agent/graph/workflow.py) + [workflow.py#L297-L299](src/multi_agent/graph/workflow.py)

**问题**: 在 review 拒绝 → 重试的工作流中，`retry_count` 在两个地方被递增：

1. `pm_review`（第 265 行）：任务被拒绝回退到 TODO 时 `task.retry_count += 1`
2. `handle_failure`（第 299 行）：PM 决定重试时 `task.retry_count += 1`

当 `route_after_review` 返回 `"max_retries"` 时，任务会进入 `handle_failure`，此时 `retry_count` 已经被 `pm_review` 递增过一次，`handle_failure` 再递增一次。这意味着每个完整的"失败→拒绝→重试"循环，retry_count 增加 2 而非 1。配置 `max_retries=3` 时，实际只允许约 1-2 次重试。

**修复**: 只在一个地方递增 retry_count。建议移除 `pm_review` 中的递增，由 `handle_failure` 统一管理：

```python
# pm_review 中（第 263-265 行）：
else:
    task.transition(TaskStatus.TODO, updated_by="pm")
    # 移除: task.retry_count += 1  — retry_count 由 handle_failure 统一管理
```

### 5. CORS 配置 `allow_origins=["*"]` + `allow_credentials=True` 存在安全风险

**位置**: [main.py#L56-L62](src/multi_agent/main.py)

**问题**: `allow_origins=["*"]` 与 `allow_credentials=True` 同时设置。虽然 Starlette 的实现会将 `Access-Control-Allow-Origin` 反射为请求来源（而非字面量 `*`），但这意味着**任意网站**都可以携带 Cookie/认证信息向本 API 发起跨域请求，生产环境存在安全隐患。

**修复**: 改为显式来源列表或通过环境变量配置：

```python
allow_origins=settings.allowed_origins.split(",") if settings.allowed_origins else ["*"],
allow_credentials=True,
```

### 7. `project_finalize` 将含 `HUMAN_PENDING` 任务的项目标记为 `FAILED`

**位置**: [workflow.py#L346-L351](src/multi_agent/graph/workflow.py)

**修复**: 利用已定义但未使用的 `ProjectStatus.PAUSED`：

```python
if failed == 0 and human_pending == 0:
    status = ProjectStatus.COMPLETED

    status = ProjectStatus.PAUSED
else:
    status = ProjectStatus.FAILED
```

## 变更摘要

- **多 Agent 工作流编排系统**：基于 FastAPI + LangGraph 构建两层编排架构（Gateway 路由 → PM 拆解 → Worker 执行 → PM 验收循环），支持即时任务和项目型任务两种模式
- **持久化层**：asyncpg + PostgreSQL，四张表（projects/tasks/trace_logs/agent_prompts），支持 JSONB 字段、GIN 索引、部分索引
- **Agent 体系**：BaseWorker 抽象基类 + Analyzer/Coder/Tester 三种 Worker 实现，统一 JSON 输出解析与容错，支持 Prompt 数据库化动态加载与版本管理
- **Gateway 路由器**：正则注入检测（中英文模式）+ LLM 意图分类，路由到 instant/project/blocked 处理链路
- **API 层**：FastAPI 提供统一聊天入口、项目/任务查询、人工介入、Trace 查询、Prompt CRUD 等 REST 接口
